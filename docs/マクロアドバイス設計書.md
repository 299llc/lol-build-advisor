# ろるさぽくん — マクロアドバイス設計書（Phase 2）

> 本書は Phase 2 で追加するマクロアドバイス機能の設計ドキュメントである。
> Phase 1 の基本設計は [基本設計書](基本設計書.md) を参照。

---

## 1. 機能概要

試合中に繰り返し更新する機能。戦況の細かい推定よりも、オブジェクトと直近イベントを中心に「今すべき最善のアクション1つ」を具体的に指示する。

### 解決する問題

ゲーム中盤以降、どこにいて何をすべきかの判断が難しい。「マップを見ろ」のような抽象論ではなく、「ドラゴンが1分後にスポーンするからボットサイドにワードを置いて、ウェーブをプッシュしておく」のような手順付きのアドバイスを提供する。

## 4. AIパイプライン

### 4.1 モデル

`Gemini 2.5 Flash-Lite` を使用する。高頻度で呼ばれるため、コスト・速度を優先する。

### 4.2 API

`Interactions API` で `macro_interaction` を試合ごとに1本作成し、`previous_interaction_id` で会話を継続する。

これにより、Phase 1 の `build_interaction`（1本）と合わせて、1試合あたり計 2 本の interaction 構成となる。

### 4.3 入力

- 初回ターンで渡す静的情報:
  自分のチャンプ、ロール、味方/敵チーム構成、プレイヤーランク、マクロ方針の前提知識
- 更新ターンで渡す動的情報:
  ゲーム時間、オブジェクト状態、現在の自分の状態、直近イベント
- 必要に応じて前回アドバイスの要約

### 4.4 コード側でやること

- オブジェクト残り時間や直近イベントを整理する
- 重い勝利条件推定や細かいレーン状態推定は行わない
- 試合開始時にマクロアドバイス用 interaction を作成し、試合中はその `previous_interaction_id` を保持する
- 初回ターンでは静的情報をまとめて投入し、更新ターンではイベント差分を中心に送る

### 4.5 Gemini に任せること

- 状況に応じた次のアクションを自由に提案する
- タイトル、説明、注意点を短く生成する
- 会話履歴を踏まえて、前回アドバイスと変える場合だけ理由を強めに説明する

### 4.6 出力

```json
{
  "action": "prepare_dragon_area",
  "title": "ドラゴン準備",
  "desc": "ボット側を先に触って視界を取る",
  "warning": "孤立して先に捕まらないよう注意",
  "confidence": "high"
}
```

必須項目は `action`, `title`, `desc`, `warning`, `confidence`。`action` は自由文字列だが、英数字と `_` を使った機械可読な識別子に統一する。

## 5. トリガーとデバウンス

| タイプ | デバウンス |
|-------|-----------|
| イベント駆動 | 30秒 |

トリガーとなるイベント:

- オブジェクト獲得直後
- オブジェクト接近（スポーン 90 秒前を目安）
- キルまたはデス
- 敵の復活
- 味方の復活
- ゴールド閾値超え
- バフ切れ
- 5 分安全弁（5分間何もイベントがない場合に1回だけ呼ぶ）

---

## 6. フォールバック

- JSON 不正なら前回アドバイスを維持する
- 初回生成前に失敗した場合は `farm` または `ward` の安全な定型文を出す

---

## 7. 試合後コーチングとの連携

Phase 2 では試合後コーチングに以下の入力が追加される。

- **マクロアドバイス履歴の要約**: 試合中にどのような提案が行われ、どの局面で方針転換があったかの短いサマリ（5〜10件程度の要点に要約）
- Gemini への追加タスク: 試合中に出していたマクロ提案と実際の試合展開を踏まえた振り返り

マクロアドバイス履歴の要約が取れない場合は、Phase 1 と同様にスナップショットとイベントログだけで生成を継続する。

---

## 8. セッション復元への追加

Phase 1 のローカル保存項目に以下を追加する。

- `macro_interaction_id`
- マクロアドバイス履歴の要約元データ

観戦モードで対象プレイヤーを切り替えた場合は、`build_interaction` と `macro_interaction` の両方を破棄して選択プレイヤー視点で作り直す。
