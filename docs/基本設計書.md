# ろるさぽくん — 基本設計書

## 前提条件

- Windows デスクトップアプリ（Electron + React + shadcn/ui + Tailwind CSS）
- LoLの試合中にリアルタイムでAIコーチングを提供する

---

# 前半：プロダクト

## 1. 5つの機能

| # | 機能 | 解決する問題 | いつ動く |
|---|------|-------------|---------|
| 1 | アイテム提案 | 敵構成に応じたアイテム選択がわからない | 試合中（状況変化時） |
| 2 | 対面マッチアップTip | 知らない対面で何に気をつけるべきかわからない | 試合開始時（ポジション確定後） |
| 3 | マクロアドバイス | ゲーム中盤以降、どこにいて何をすべきかわからない | 試合中（定期的） |
| 4 | ルールベースアラート | ファーム効率低下やオブジェクトタイミングを見逃す | 試合中（常時） |
| 5 | 試合後コーチング | 何が良くて何が悪かったかわからない | 試合終了後 |

---

## 2. 画面設計

**設計原則: 試合体験を邪魔しない。** ウィンドウは常時最前面だが、LoLの操作を邪魔しないサイズ・位置にする。CPU/メモリを食いすぎてゲームのFPSを落とさない。情報は短く。読ませるのではなく「見せる」。

### 2.1 フェーズに応じた画面切替

LoLの試合フローに合わせて画面を切り替える。

```
待機中 ──► チャンプセレクト ──► 試合中 ──► 試合終了
                                          │
                                          └──► 待機中（次の試合へ）
```

| フェーズ | 表示内容 |
|---------|---------|
| 待機中 | 接続状態、設定画面（プロバイダー選択・ランク設定） |
| チャンプセレクト | チーム構成、ルーン・スペル情報（LCU APIから取得） |
| 試合中 | **5機能のうち4つ**（アイテム提案、マッチアップTip、マクロアドバイス、アラート） |
| 試合終了 | 試合後コーチング |

### 2.2 試合中画面のレイアウト方針

「チラ見でわかる」が最重要。ゲーム中に読み込む余裕はない。

- **1つのアドバイスは2-3行以内**に収める
- 優先度の高い情報を上に、補足は折りたたみ
- 常時最前面のオーバーレイとしてゲーム画面の邪魔にならないサイズ

主要パネル構成：

```
┌──────────────────────────┐
│ タイトルバー（ドラッグ移動）│
├──────────────────────────┤
│ スコアボード（両チーム）    │
├──────────────────────────┤
│ ビルド提案（機能1）        │
├──────────────────────────┤
│ マッチアップTip（機能2）   │
├──────────────────────────┤
│ マクロアドバイス（機能3）   │
├──────────────────────────┤
│ アラート（機能4）          │
└──────────────────────────┘
```

### 2.3 コンパクトビュー（オーバーレイ小窓）

試合中にメインウィンドウを最小化した状態でも、最低限の情報をゲーム画面上に表示するための透過オーバーレイ。

- **背景透過**: ゲーム画面の上に半透明で重ねて表示。視界を遮らない
- **表示内容**:
  - コアビルドアイテム（購入済みはチェック付きで半透明化）+ AI提案アイテム（最大3個）
  - スキルオーダー（上げ順を右寄せで表示）
  - マクロアドバイス（1行タイトル＋短い説明、ゲーム内時間付き）
- **操作**:
  - **ロック状態**（デフォルト）: ウィンドウがクリックスルーになり、ゲーム操作を一切邪魔しない。ロックアイコンのみホバーで操作可能
  - **アンロック状態**: ドラッグで位置移動・リサイズが可能。閉じるボタンでコンパクトビューを終了
- **切り替え**: メインウィンドウのボタンまたはショートカットキーでコンパクトビューに切り替え
- **ステータスインジケーター**: 左上の小さな丸で状態を表示（試合中=青、試合終了=赤、待機中=金色で点滅）

```
┌──────────────────────────────────┐
│ ●                         🔒    │
│ [アイテム1][2][3] | [AI1][AI2]  Q>W>E │
│ 🎯 MACRO                  12:30 │
│ ドラゴン準備 - BOT側に寄る        │
└──────────────────────────────────┘
```

### 2.4 観戦モード

LoLの観戦機能で他プレイヤーの試合を視聴しているとき、選択したプレイヤーの視点でAI分析を行う**開発者用デバッグ機能**。

一般ユーザー向け機能としては提供しない。リリース版の UI には露出せず、開発・検証時のみ有効化する前提とする。

#### 検出方法

Live Client Data APIの `activePlayer` フィールドが空（観戦時は自分がプレイヤーではないため空になる）のとき、自動的に観戦モードとして判定する。特別な設定は不要。

#### 画面と操作（デバッグ時のみ）

- 観戦モード検出時、画面上部に「SPECTATOR」ラベルとプレイヤー選択ドロップダウンを表示
- ドロップダウンには全プレイヤーがチャンピオン名＋サモナー名で一覧表示される
- プレイヤーを選択すると、そのプレイヤーを「自分」として扱い、コアビルド・マッチアップTip・AI提案などをすべて再取得・再生成する

#### 動作仕様

- プレイヤー選択時にビルド状態を全リセットし、次のポーリングで新しいプレイヤー視点のデータを再取得する
- すべてのAI機能（アイテム提案・マッチアップTip・マクロアドバイス・コーチング）が選択プレイヤー視点で動作する
- ルールベースアラートも選択プレイヤー基準で評価される
- 無料枠・有料枠の試合数カウント対象には含めない

## 3. データの時系列フロー

```
1. アプリ起動
   → 待機画面。LoLクライアントの接続を監視

2. チャンプセレクト検出
   → チーム構成を表示

3. 試合開始
   → 静的データ取得（チャンプ情報、コアビルド）
   → ポジション確定後、対面チャンプを特定 → マッチアップTip生成（AI、1回）
   → 試合ログ蓄積を開始（コーチング用）
   → ポーリング開始

4. 試合中ループ（数秒間隔）
   → ゲームデータ取得
   → 60秒間隔でスナップショットを試合ログに蓄積（コーチング用）
   → ルールベースアラート評価（即時）
   → 差分検知 → 条件を満たせばAI呼び出し
      → アイテム提案更新
      → マクロアドバイス更新

5. 試合終了
   → 蓄積した試合ログからコーチング用の集計データを生成
   → 試合後コーチング生成（AI、1回）
   → 結果表示 & ローカル保存

6. → 1に戻る
```

# 後半：アーキテクチャ

## 4. システム全体像

```
┌───────────────────────────────────────┐
│          ろるさぽくん (Electron)        │
│                                       │
│  ┌───────────┐  IPC  ┌────────────┐  │
│  │ Renderer   │◄─────►│   Main     │  │
│  │ (React UI) │       │ (Backend)  │  │
│  └───────────┘       └─────┬──────┘  │
└────────────────────────────┼─────────┘
                             │
           ┌─────────────────┼─────────────┐
           │                 │             │
  ┌────────▼──────┐  ┌──────▼─────┐ ┌─────▼──────┐
  │ LoLクライアント │  │ AIサービス  │ │ ビルドデータ │
  │ (試合データ)    │  │ (判断生成)  │ │ (統計情報)   │
  └───────────────┘  └────────────┘ └────────────┘
```

アプリは大きく **「データを集める」→「判断を作る」→「画面に出す」** の3段階で動く。

Main Processの役割はデータ収集・判断生成・アプリ管理（ウィンドウ管理、設定の永続化等）の3つ。

### 4.1 外部依存

| 依存先 | 何を取るか | いつ使えるか |
|--------|-----------|-------------|
| Live Client Data API (`localhost:2999`) | 試合中のリアルタイムデータ | 試合中のみ |
| LCU API | チャンプセレクト情報、ゲームフロー | LoLクライアント起動中 |
| AI（Gemini / Ollama） | アドバイス生成 | 常時（Ollamaはローカル） |
| OP.GG等の統計サイト | コアビルド、カウンターアイテム | 常時 |
| Data Dragon（Riot CDN） | チャンプ名・アイテム名・画像 | 常時 |

### 4.2 フリーミアムモデルと動作モード

ビジネスモデルはフリーミアム。ただし区別は「ローカルかクラウドか」ではなく、**Gemini の利用回数制限**で行う。

初期リリースでは Gemini を本線とし、無料プランでは **1日あたりの試合数** に上限を設ける。上位プランではその上限を引き上げる。ローカル/Ollama は将来のために残すが、プロダクトの主導線としては扱わない。

| プラン | AI経路 | 品質 | 制限 | 認証 |
|--------|--------|------|------|------|
| **無料枠** | Google Gemini | 高。Gemini 2.5 Flash / Flash-Lite | 1日3試合まで | ビルド時に埋め込み |
| **有料枠** | Google Gemini | 高。Gemini 2.5 Flash / Flash-Lite | 試合数上限を拡張、または実質無制限 | ビルド時に埋め込み |
| **ローカル補助** | Ollama（ユーザーPC上） | 低〜中。小型モデルの限界あり | 将来向け。主導線にはしない | 不要 |

無料枠のカウントは **1試合終了ごとに 1 カウント** とする。試合中に Gemini を何回呼んだかでは数えない。

- `remake` はカウントしない
- `surrender` はカウントしない
- 通常終了した試合のみ 1 カウントとする
- 無料枠上限に達した場合でも、**進行中の試合は最後まで継続**し、次の新規試合開始時から AI 機能を停止する

通常終了の判定は、**ネクサスタワーが折れた時点で `normal_end=true`** とする。このフラグが立った試合のみ、試合終了処理と無料枠カウントの対象にする。

#### クラウドモードのアーキテクチャ

```
Electron App  ──REST API──>  Gemini API
               (APIキー認証)
```

- Google Gemini APIをメインプロセスから直接呼び出す
- 高頻度機能（アイテム提案・マクロアドバイス）は `Gemini 2.5 Flash-Lite` を優先し、`thinkingBudget=0` でコスト・速度を最適化する
- 単発機能（マッチアップTip・試合後コーチング）は `Gemini 2.5 Flash` を利用し、品質を優先する
- サーバーサイドのインフラ（API Gateway / Lambda）は不要
- `response_schema` でJSON出力スキーマを強制し、後処理のバリデーション負荷を軽減
- **Gemini 2.5 Flash-Lite**: 低コスト・高頻度向け。アイテム提案・マクロアドバイスに使用
- **Gemini 2.5 Flash**: 品質重視の単発処理向け。マッチアップTip・試合後コーチングに使用

#### Gemini認証情報の管理

- 開発者のGemini APIキーをビルド時に `.env` で埋め込む
- 認証情報はElectronのメインプロセス側で保持し、Renderer側には露出させない
- ⚠️ ElectronアプリへのAPIキー埋め込みはasarの展開等でキーが抽出されるリスクがある。初期段階では許容するが、将来的には自前APIサーバー経由への移行を検討する

#### 設定画面でのプロバイダー選択

設定画面では Gemini を既定の AI 経路として扱い、現在のプランで使える **残り試合数** や上限状況を表示する。必要に応じて「省コスト優先（Flash-Lite中心）」と「品質優先（Flash中心）」のプリセットを選べるようにする。ローカル/Ollama は詳細設定内の補助項目として残すが、初回導線には置かない。

無料枠が 1 試合分以下になった場合は設定画面と待機中画面に警告を表示する。無料枠を使い切った後は、進行中試合の AI は継続し、次の新規試合開始時に「本日の無料枠を使い切りました」と表示して AI 機能を停止する。

### 4.3 エラーハンドリング

- API呼び出しやLLMのタイムアウト時は、UIに「うまく取得できませんでした」と表示するだけでよい
- 前回の正常な出力があればそれを維持し、新しいエラー表示で上書きしない
- ゲーム体験を壊さないため、エラーでアプリがクラッシュしたりフリーズしたりしないことを最優先とする

## 5. データ収集

LoLクライアントが試合中に公開するローカルAPI（Live Client Data API）から、数秒間隔でゲームデータを取得する。

取得できる情報：
- 全プレイヤーのチャンピオン・レベル・アイテム・KDA・CS
- 自プレイヤーの現在ゴールド
- ゲーム内時間
- イベントログ（キル、オブジェクト獲得等）

加えて、LCU（League Client Update）APIからチャンプセレクト中のピック情報・ルーン・サモナースペルも取得する。

### 5.1 対面チャンピオンの判定

チャンプセレクト時点では敵のロール割り当てが確定しない（フレックスピック・ロールスワップがあり得る）。Live Client Data APIは試合開始後に各プレイヤーのポジション情報を返すため、**対面の特定は試合開始時に行う**。マッチアップTipの生成もこのタイミングで実行する。

### 5.2 Live Client Data API で取得できないもの

- **他プレイヤーのゴールド**: 自分以外のゴールドは取得不可。アイテム価格の合算で推定する
- **レーン状態（laneState）**: APIにはウェーブ位置の情報がない。直近のキル/デス/タワー破壊イベントの発生レーンから簡易推定する。精度は低いため、マクロアドバイスでは補助情報として扱う
- **ドラゴン取得チーム**: イベントログのDragonKillイベントにKillerNameが含まれるため、プレイヤーのチームと照合して判定する

## 6. AIパイプライン

**設計原則:**
- **Gemini を本線にする**: 全プランで Gemini を前提にし、ローカル/Ollama は将来向けの補助経路として扱う
- **前処理は最小限にする**: 重い推定エンジンではなく、入力整形・候補制約・トリガー制御に限定する
- **入出力は固定 JSON**: すべての AI 機能で固定スキーマを使い、自由回答を UI に直接出さない
- **イベント駆動で呼ぶ**: 定期実行は最小限に抑え、意味のある変化があった時だけ Gemini を呼ぶ

### 6.1 共通原則

4機能はそれぞれ別のパイプラインとして設計する。ただし共通で守るルールは同じ。

- コード側がやること:
  取得済みデータの正規化、必要フィールドの抽出、候補の制約、呼び出しタイミングの制御
- Gemini に任せること:
  事実の整理、提案内容の決定、短い理由生成、プレイヤーに見せる自然な文章化
- UI に渡す前にやること:
  JSON スキーマ検証、文字数圧縮、前回値維持またはテンプレートフォールバック

#### 共通入力の考え方

全機能で共通の巨大な試合状態オブジェクトは持たず、**取得事実を中心にした最小入力**を機能ごとに組み立てる。

残す情報:
- 自分のチャンプ、ロール、レベル、KDA、CS、所持アイテム、現在ゴールド
- 対面チャンプまたは敵主要チャンプ
- ゲーム時間、オブジェクト状態、直近イベント
- プレイヤーランク

主設計から外す重い推定:
- 味方/敵チームの推定ゴールド
- 細かいレーン状態推定
- 過剰な構成タグ分類
- スキル説明からの大量タグ抽出

#### モデルの使い分け

- `Gemini 2.5 Flash-Lite`:
  高頻度のアイテム提案、マクロアドバイス
- `Gemini 2.5 Flash`:
  単発で品質を重視するマッチアップTip、試合後コーチング

#### API の使い分け

- **ビルド提案 / マクロアドバイス**:
  `Interactions API` を使い、試合ごとに 1 つの interaction を継続する
- **マッチアップTip / 試合後コーチング**:
  単発処理のため `generateContent` を使う

試合中の継続系では、試合開始時に interaction を作成し、以後は `previous_interaction_id` で会話を継続する。これにより、前回アドバイスとの一貫性を API 側の履歴で維持しやすくする。

継続系の最適化方針は **「初期ターンに静的情報、その後は動的差分だけ送る」** とする。試合開始時にその試合で変わらない情報を interaction に入れ、更新ターンでは現在状態と候補だけを追加する。

#### 1試合あたりの interaction 構成

1試合で保持する interaction は **2本** に固定する。

- `build_interaction`:
  ビルド提案専用
- `macro_interaction`:
  マクロアドバイス専用

マッチアップTip と試合後コーチングは単発生成のため interaction を持たない。観戦モードで対象プレイヤーを切り替えた場合は、該当試合の 2 本の interaction を破棄して選択プレイヤー視点で作り直す。

アプリがクラッシュした場合でも、同一試合中であれば保存済みの `interaction_id` と試合ログから同じセッションに復帰する。復帰後は新規試合作成ではなく、既存の `build_interaction` / `macro_interaction` を再利用する。

復元のためにローカル保存する最小項目は以下とする。

- `game_id`
- `game_start_time`
- `build_interaction_id`
- `macro_interaction_id`
- 現在のプレイヤー識別子
- 試合ログのスナップショット
- マクロアドバイス履歴の要約元データ

これらは試合終了後コーチングの生成完了時、または通常終了を確認して一定時間経過後に削除する。

ローカル/Ollama は同じ入出力スキーマを維持した補助経路として残すが、日常利用の本線としては想定しない。

### 6.2 マッチアップTip

試合開始直後に 1 回だけ生成する。細かい推定は行わず、対面確定とチャンプ情報の整理だけをコード側で担当する。

#### 入力
- 自分のチャンプ、ロール
- 対面チャンプ
- Data Dragon の基本情報:
  スキル名、パッシブ名、スキル説明、推奨される役割
- プレイヤーランク

#### コード側でやること
- 試合開始後に対面を確定する
- Data Dragon から必要フィールドだけ抽出する
- Gemini に渡す JSON を短く整形する
- スキル説明全文をそのまま渡す場合も、不要な装飾文は除去する

#### Gemini に任せること
- 危険タイミングの整理
- 意識すべきポイントの要約
- 短い対面ガイドの生成

#### 出力

```json
{
  "summary": "互角〜やや不利",
  "tips": ["スタンの溜まり具合を見てから前に出る", "Lv6以降のオールインに注意", "スキル使用後の隙に短く返す"],
  "danger": "相手の主要CCが残っている時の前進",
  "power_spike": "Lv6"
}
```

必須項目は `summary`, `tips`, `danger`, `power_spike`。`tips` は 1〜3 件の短文配列とする。

#### トリガー
- 試合開始後、ポジション確定時に 1 回

#### フォールバック
- Gemini が失敗した場合はテンプレート文で簡易表示する
- 例:
  `danger = 相手の主要スキルに注意`

### 6.3 ビルド提案

試合中に繰り返し更新する機能。存在しないアイテムや極端な提案を防ぐため、候補制約はコード側で残す。

#### 入力
- 初回ターンで渡す静的情報:
  自分のチャンプ、ロール、敵チーム構成、コアビルド情報、プレイヤーランク
- 更新ターンで渡す動的情報:
  現在ビルド、所持ゴールド、敵チームの主要脅威、敵の回復持ち有無、CC の多さ、ダメージ傾向、候補アイテム一覧
- 必要に応じて前回提案の要約

#### コード側でやること
- 候補アイテムを最大 5 件程度まで絞る
- 候補は**完成品のみ**を対象とし、素材アイテムは候補に含めない
- 候補には `core / counter / situational` など最低限の理由タグを付ける
- 候補外のアイテムが選べないようにする
- 試合開始時にビルド提案用 interaction を作成し、試合中はその `previous_interaction_id` を保持する
- 初回ターンでは静的情報をまとめて投入し、更新ターンでは差分だけ送る

#### Gemini に任せること
- 候補から最大 3 件を選ぶ
- 理由と次の購入方針を短く生成する
- 会話履歴を踏まえて、前回提案から変える場合は変更理由を説明する

#### 出力

```json
{
  "recommended": [
    { "id": 3156, "reason": "敵APが脅威のためMRを優先" }
  ],
  "reasoning": "次のリコールではMR付きの完成を優先",
  "confidence": "high"
}
```

必須項目は `recommended`, `reasoning`, `confidence`。`recommended` は 1〜3 件、各要素は `id`, `reason` を必須とする。

#### トリガー
- アイテム購入
- キルまたはデス
- 敵脅威や候補アイテムが変わった時
- デバウンスは 90 秒

#### フォールバック
- JSON 不正または候補外の提案なら前回提案を維持する
- 初回生成前に失敗した場合はコアビルドをそのまま表示する

### 6.4 マクロアドバイス

試合中に繰り返し更新する機能。戦況の細かい推定よりも、オブジェクトと直近イベントを中心に次アクションを決める。

#### 入力
- 初回ターンで渡す静的情報:
  自分のチャンプ、ロール、味方/敵チーム構成、プレイヤーランク、マクロ方針の前提知識
- 更新ターンで渡す動的情報:
  ゲーム時間、オブジェクト状態、現在の自分の状態、直近イベント
- 必要に応じて前回アドバイスの要約

#### コード側でやること
- オブジェクト残り時間や直近イベントを整理する
- 重い勝利条件推定や細かいレーン状態推定は行わない
- 試合開始時にマクロアドバイス用 interaction を作成し、試合中はその `previous_interaction_id` を保持する
- 初回ターンでは静的情報をまとめて投入し、更新ターンではイベント差分を中心に送る

#### Gemini に任せること
- 状況に応じた次のアクションを自由に提案する
- タイトル、説明、注意点を短く生成する
- 会話履歴を踏まえて、前回アドバイスと変える場合だけ理由を強めに説明する

#### 出力

```json
{
  "action": "prepare_dragon_area",
  "title": "ドラゴン準備",
  "desc": "ボット側を先に触って視界を取る",
  "warning": "孤立して先に捕まらないよう注意",
  "confidence": "high"
}
```

必須項目は `action`, `title`, `desc`, `warning`, `confidence`。`action` は自由文字列だが、英数字と `_` を使った機械可読な識別子に統一する。

#### トリガー
- オブジェクト獲得直後
- オブジェクト接近（スポーン 90 秒前を目安）
- キルまたはデス
- 敵の復活
- 味方の復活
- ゴールド閾値超え
- バフ切れ
- 5 分安全弁
- デバウンスは 30 秒

#### フォールバック
- JSON 不正なら前回アドバイスを維持する
- 初回生成前に失敗した場合は `farm` または `ward` の安全な定型文を出す

### 6.5 試合後コーチング

試合終了後に 1 回だけ生成する。ここだけは試合中の蓄積と集計が必須。

#### 入力
- 60 秒間隔で蓄積したスナップショット
- イベントログ
- 集計値:
  KDA 推移、CS 推移、アイテム購入順、オブジェクト参加、フェーズ別成績
- マクロアドバイス履歴の要約:
  試合中にどのような提案が行われ、どの局面で方針転換があったかの短いサマリ
- プレイヤーランク

#### コード側でやること
- 試合中のスナップショットをメモリに蓄積する
- 試合終了時に集計データへ変換する
- マクロアドバイスの interaction 履歴を 5〜10 件程度の要点に要約する
- 数値や時系列をそのまま Gemini に渡せる JSON に整形する

#### Gemini に任せること
- 試合全体の総括
- 良かった点と改善点の抽出
- 試合中に出していたマクロ提案と実際の試合展開を踏まえた振り返り
- 次試合への短いアドバイス生成

#### 出力

```json
{
  "overall_score": 7,
  "sections": [
    { "title": "レーン戦", "content": "CSは安定していたが、デス後の復帰が遅かった", "grade": "B" }
  ],
  "good_points": ["CS維持", "オブジェクト前の寄り"],
  "improve_points": ["デス後の復帰判断", "リコールタイミング"],
  "next_game_advice": "オブジェクト60秒前の準備を早める"
}
```

必須項目は `overall_score`, `sections`, `good_points`, `improve_points`, `next_game_advice`。`sections` は 1 件以上、各要素は `title`, `content`, `grade` を必須とする。

採点基準は以下で固定する。

- `overall_score`:
  1〜10 の整数
- `grade`:
  `S / A / B / C / D`
- 評価軸:
  レーン戦、ファーム、デスの重さ、オブジェクト関与、ビルド妥当性
- ランク補正:
  同じ内容でもプレイヤーランクに応じて期待値を調整する
  例: Bronze と Gold では CS やデス数の評価基準を変える

#### トリガー
- 試合終了後に 1 回

#### フォールバック
- Gemini が失敗した場合は集計値だけの簡易レポートを表示する
- マクロアドバイス履歴の要約が取れなくても、スナップショットとイベントログだけで生成を継続する
- スコアやグレードが不正なら、数値評価なしのテキスト版に切り替える

### 6.6 共通フォールバックと呼び出し制御

#### 共通フォールバック

- JSON スキーマに合致しない場合は UI に出さない
- 前回値がある機能は前回値を維持する
- 初回で前回値がない機能はテンプレート表示に落とす
- タイトル、説明、理由は UI で読める長さに圧縮する
- 抽象的すぎる一般論は抑制する
- Interactions API が失敗した場合でも、そのターンだけ通常の単発呼び出しにフォールバックできる形にする
- アプリ再起動後に同一試合を検出した場合は、保存済みセッションの復元を最優先する

#### スキーマ厳格化ルール

- すべての AI 出力は `additionalProperties: false` 相当で余計なキーを許可しない
- `confidence` は `high / medium / low` の列挙型に固定する
- 文字数制限を超える出力は後処理で切り詰める
- 必須キーが欠けた場合はそのレスポンスを破棄し、フォールバックに切り替える

#### トリガーとデバウンス

| 機能 | タイプ | トリガー | デバウンス |
|------|-------|---------|-----------|
| マッチアップTip | 単発 | 試合開始時（ポジション確定後） | なし |
| 試合後コーチング | 単発 | 試合終了時 | なし |
| ビルド提案 | イベント駆動 | アイテム購入 / キルデス / 候補や状況の変化 | 90秒 |
| マクロアドバイス | イベント駆動 | オブジェクト獲得・接近 / キルデス / ゴールド閾値超え / バフ切れ / 5分安全弁 | 30秒 |
| ルールベースアラート | 常時 | ポーリングごと | なし（AI不要） |

#### 変化検知

意味のある変化があった場合のみ Gemini を呼ぶ。

- アイテム購入
- キルまたはデス
- オブジェクト獲得
- オブジェクト接近
- ゴールド閾値超え
- バフ切れ
- 試合開始
- 試合終了

定期呼び出しは行わず、5 分間何もイベントがない場合のみ安全弁として 1 回だけ呼ぶ。

#### 試合終了判定

- `normal_end=true`:
  ネクサスタワーが折れた時点で立てる
- `normal_end=true` の場合:
  試合後コーチングを生成し、無料枠カウント対象とする
- `remake` / `surrender`:
  試合終了画面には遷移しても、無料枠カウント対象にはしない
